- [x] 4.4 Implement "Question-answering endpoint supports server-sent event streaming" so `?stream=true` returns `text/event-stream` whose every `data:` line is a single JSON document, with the documented event order (one `transcript`, then zero or more `token` events, terminating `done`); the contract also covers the "Streaming error before transcript (transcription failure)" and "Streaming with empty LLM response" scenarios from the `ask` capability (the latter is the only success path with zero `token` events); verify with `tests/test_ask.py` parsing the SSE stream against mocked streaming LLM and STT responses for the success path, the LLM-error-after-transcript path, the STT-failure-before-transcript path, and the empty-LLM-response path.

## 5. New `WS /listen` WebSocket streaming endpoint

- [ ] 5.1 Implement the design decision "Add `WS /listen` with timestamped partial and final events" plus the spec requirement "WebSocket endpoint accepts 16 kHz mono PCM audio" by creating `app/api/listen.py` and `app/services/stream.py` containing the embedded sliding-window VAD wrapper described in the design decision "Embed a sliding-window streaming wrapper instead of depending on an upstream `whisper-streaming` package"; the wrapper SHALL convert `pcm_s16le` frames to `float32` internally and feed them to the shared `WhisperModel` in 500 ms re-transcription windows; verify with `tests/test_listen.py` opening the socket, streaming a recorded `pcm_s16le` sample (frame size 8 KB ≈ 250 ms of 16 kHz mono `pcm_s16le`), and asserting chunked ingestion (not whole-file buffering).
- [ ] 5.2 Implement "Server emits timestamped partial and final transcript events" so each event matches `{"type": "partial" | "final", "text", "start_ms", "end_ms"}` with timestamps relative to stream start and monotonically non-decreasing across utterances within the same connection; cover the "Multiple utterances per connection" scenario by feeding two utterances separated by silence and asserting two `final` events back to back; verify with `tests/test_listen.py`.
- [ ] 5.3 Implement "Disconnect mid-utterance discards in-flight buffer" so closing the socket during a partial stream stops processing without emitting a `final` and so the "Client closes after a final event but before sending more audio" scenario does not emit any further events; verify with `tests/test_listen.py` covering both close paths plus the INFO-level log assertion.
- [ ] 5.4 Implement the PCM frame size guards: reject non-binary frames, frames < 200 bytes, and frames > 64 KiB with `{"type":"error","message":"..."}` and a WebSocket close code `1003`; verify with `tests/test_listen.py` cases for text frame, undersized binary, oversized binary.
- [ ] 5.5 Implement the "Server applies backpressure when audio arrives faster than ASR consumes" requirement: cap the in-flight PCM buffer at 30 seconds, drop oldest audio on overflow, and emit one `{"type":"warning","message":"buffer overflow, oldest audio dropped"}` per overflow event without closing the connection; verify with `tests/test_listen.py` simulating fast client + slow ASR mock and asserting the warning frame plus continued operation.
